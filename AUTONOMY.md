# 運用ランブック（手動セッション・自動セッション共通）

このファイルは money-tools を改善する**すべての Claude セッションが従う唯一のルールブック**です。

- **手動セッション**（あなたが Claude Code に指示出ししながら進める）
- **自動セッション**（クラウドのスケジュール実行が無人で進める）

どちらも**毎回ここを読み、同じ手順・同じ優先順位・同じガードレールで作業する**。
迷ったら「正確さ（YMYL）を守りつつ、検索流入と収益を増やす」に従う。

> 収益目標：まず **Claude Code のサブスク代を月々まかなう** → **月1万円** → **月10万円**。
> AdSense + アフィリ収益はほぼ「**検索流入 × ページの質 × 収益化の最適化**」で決まる。

---

## 🔑 大原則（手動・自動 共通／最優先で守る）

1. **取り込みは PR 経由が既定**。`auto/*` ブランチ → PR → CI(ビルド)緑 → 自動マージ。
   - 手動セッションでも、まとまった変更は同じく PR で入れることを推奨（ルール統一）。
   - 例外: ごく軽微な手動修正を直接 main に入れる場合も、**必ず `npm run build` が通る状態**で。
2. **`src/lib/tax.ts`（税率・控除）もフルオートで自動マージ**。安全弁は **`src/lib/calc.test.ts`（既知の公表値レンジ＋不変条件）が CI で必ず通ること**＝壊れた計算は自動ブロック。変更時は `money-logic-changed` ラベルが付き事後確認できる。**テストを壊れた値に合わせて緩めない**。料率は公式値のみ。
3. **1実行 = 1改善**。壊れた状態を残さない・push しない。
4. **重複を作らない**。着手前にバックログ・既存ページ・open PR を確認。
5. **並行作業の衝突を避ける**：
   - 自動作業は**クラウド（隔離環境）**で動かす。ローカルの手動セッションとワークツリーを共有しない。
   - **同じローカル作業ツリーで2つのセッションを同時に走らせない**（`git add -A` の巻き込み・ブランチ競合の原因）。
   - 手動で別ブランチ作業をしたい時は `git worktree` を使って隔離する。

---

## 🎯 優先順位（この順で着手する）

**収益はデータ駆動で最大化する**：まず計測し、流入・収益が伸びる施策に寄せ、伸びなければ見切る。
ヘッドワード（手取り計算/ふるさと納税 等）は公式サイトが独占し新規ドメインでは上位困難。
だから**①低競合の長尾を量産して流入を取り、②流入非依存の有料ツールも試す**という複線で進める。
毎回この順で「未完で最も価値が高い1件」を選ぶ。上が枯れたら下へ。

1. **正確性・鮮度維持** — 税率/料率が古いと金融ツールとして即・無価値（YMYL）。
   `node scripts/check-freshness.mjs` を確認し、期限超過や改定期（社会保険=4月 / 税制=年末→翌年）なら最優先。→ 下の「定期レビュー」。
2. **計測の確保** — アクセス解析(Cloudflare Web Analytics)が動いているか。流入・人気ページ・離脱が見えないと改善が勘になる。→ バックログ「計測」。
3. **長尾SEO量産** — 公式が手薄な粒度ページ（年収別/自治体別/職業別の早見・解説）を量産し低競合の長尾で流入。既存ページのFAQ・内部リンク強化も。→ バックログ「SEO」。
4. **収益最適化** — 広告配置・アフィリCTA・回遊導線で、同じ流入からの収益(RPM)を上げる。→ バックログ「収益」。
5. **有料ツール/新ツール** — 流入依存の低い有料MVP（確定申告/年収の壁）や新規入口。→ バックログ「新ツール/有料」。

## 🤖 1回の実行手順（ランブック）

1. `git pull` で最新化。`node scripts/check-freshness.mjs` を実行し、鮮度切れなら**優先度1へ**。
2. 上の優先順位に従い、下の **バックログ** から未完(`[ ]`)の最優先項目を **1つだけ** 選ぶ。
3. **作業ブランチを切る**：`git switch -c auto/<slug>`（例 `auto/seo-tedori-hayami`）。
4. 実装する：
   - 計算ロジックは `src/lib/` に**純粋関数**で実装。実装直後に概算値とのズレを簡易チェック：
     `node --experimental-strip-types <一時テスト.mjs>`（確認後その一時ファイルは消す）。
   - ページは `src/pages/tools/<slug>.astro`。`tools/tedori.astro` を雛形にし、**必ず**踏襲：
     `Base` レイアウト / `path` / `description` / JSON-LD(BreadcrumbList + FAQPage) / 「概算についての注意」/ 末尾に内部リンク。
   - 新ツールなら `src/pages/index.astro` の一覧で該当項目を `ready: true` にする。
5. `npm run build` が**通ること**を確認（通らなければ直してから次へ）。壊れた状態を push しない。
6. バックログに `[x]` を付け、**ブランチを push して PR を作る**：
   ```sh
   git add -A && git commit -m "<種別>: <内容>"   # 例 "SEO: 手取りに年収別早見表を追加"
   git push -u origin HEAD
   gh pr create --fill --label auto-merge          # 自動マージ対象の目印
   ```
7. あとは **CI(`npm test`＋ビルド) と auto-merge ワークフローが自動で取り込む**（下の「取り込みの仕組み」）。
   - `src/lib/tax.ts` を触る PR も、計算テストが通れば自動マージされる（`money-logic-changed` ラベルが付く）。
8. 1実行=1改善で終了。次回また続きから。

## 🔀 取り込みの仕組み（PR → 自動マージ）

- すべての自動改善は **ブランチ `auto/*` → PR** で入れる（自動セッションは main 直コミットしない）。
- `.github/workflows/ci.yml` … 全PRで **`npm test`（計算ユニットテスト）＋ `npm run build`** を検証（誤計算・破損を止める品質ゲート）。
- `.github/workflows/auto-merge.yml` … CIが**緑**になったら自動でsquashマージ（**フルオート**）。
  **`src/lib/tax.ts` を変更するPRも自動マージ**するが、計算テストが安全弁。変更時は `money-logic-changed` ラベルが付き事後確認できる。
- 原則すべて無人で公開まで進む。`money-logic-changed` の付いた PR は事後に目を通すとよい。

## 🛡 ガードレール（絶対に守る）

- **お金をマイナスにしない**：有料サービスを勝手に契約しない。インフラは無料枠
  （Cloudflare Pages 無料・取得済みドメイン・GitHub Actions 無料枠）のみ。課金が要る施策は提案として書くだけ。
- **正確性が最優先（YMYL）**：税率・保険料率は推測で書かない。国税庁・協会けんぽ等の**公式値を確認**してから反映。
  不確かなら「概算」と明記し保守的に。`tax.ts` 変更時は **`npm test`（calc.test.ts）通過が必須**＝壊れた計算は自動ブロック。**テストを誤った値に合わせて緩めない**。
- **日本の法律・規約を順守**：誇大表現・無断転載・スパム的大量投稿・自動アカウント作成はしない。
- **1実行=1改善**に留め、必ずビルドが通る状態でPR。既存ツールの計算結果を変えたら簡易チェックで再確認。

- [x] 各ツールに「年収別 早見表」を追加（例 手取り: 300/400/500/600/700/800/1000万）。（tedori 追加済み。他ツールは次回以降）
- [ ] 各ツールに「よくある質問(FAQ)」を3〜5問追加し JSON-LD の FAQPage を充実（リッチリザルト狙い）。
- [ ] ツール間の内部リンクを強化（関連ツールへの導線をページ末尾に統一配置）。
- [ ] 各ツールの title/description を検索意図に合わせて最適化（「○○ 計算」「○○ シミュレーション」等）。
- [ ] OGP 画像（ツールごと）を用意してSNSシェアのCTRを上げる。

## 🩺 定期レビュー：料率・税制の鮮度チェック（最重要・自動化対象）

税率・社会保険料率は毎年改定される（**社会保険＝毎年4月**、**税制改正＝年末大綱→翌年施行**）。
`scripts/check-freshness.mjs` と `src/lib/freshness.ts` の `TAX_DATA_META`（`fiscalYear` / `lastReviewed` / `reviewDue`）で管理する。

**いつ:** 毎回の実行で `check-freshness.mjs` を確認。特に **3〜4月**（社会保険改定）・**12〜1月**（税制改正）は重点。

**1回のレビュー手順:**
1. 現在日を確認。`reviewDue` 超過／改定期なら重点的に。
2. 公式値をWeb検索で取得し `src/lib/tax.ts` の `RATES`・控除関数と突き合わせる：
   - 協会けんぽ 東京：当年度の健康保険料率・介護保険料率・**子ども子育て支援金**率（`https://www.kyoukaikenpo.or.jp/`）
   - 雇用保険料率（当年度・一般の事業の労働者負担）
   - 国税庁：当年分の基礎控除（合計所得別）・給与所得控除の最低保障（`https://www.nta.go.jp/`）
3. **差異があれば**：`tax.ts` の値・控除関数を更新 → 各ページの「令和◯年／◯年」表記も更新 →
   `freshness.ts` の `fiscalYear`・`lastReviewed`（今日）・`reviewDue`（翌年4/1）を更新。
   → `tax.ts` を含む PR は計算テストが通れば自動マージ（`money-logic-changed` ラベル）。
4. `npm test`（calc.test.ts）と `npm run build` の通過を確認。必要なら calc.test.ts のレンジも妥当に更新（壊れた値に合わせない）。
5. PR作成。**差異が無ければ** `lastReviewed` だけ今日に更新して PR（「確認済み」の記録。tax.ts を触らないので自動マージされる）。
6. **不確かなら変更しない**（保守的に）。公式の確定値が取れるまで待ち、必要ならバックログに残す。

**ガードレール:** 公式情報源のみ。推測で料率を書かない。ビルドが通る状態でのみ PR。

## 🔁 実行基盤

### A. クラウドのスケジュール実行（本命・無人・Mac の電源と無関係）
Claude Code のスケジュール機能（routine / cron）で「このリポジトリで AUTONOMY.md を1件実行」を毎晩走らせる。
クラウドは**隔離環境**なので、あなたのローカル手動セッションと衝突しない。
1回が利用上限に達しても、**次回のスケジュール実行で続きから再開**する。

### B. ローカルの手動セッション（あなたが指示出しする時）
このランブックに従って進める。**自動セッション(クラウド)と同時に同じツリーを触らない**こと（上の「並行作業の衝突を避ける」）。
