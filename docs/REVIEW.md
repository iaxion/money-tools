# レビュー方針（自律システムの品質ゲート）

レビューには **2種類** あり、対象で使い分ける。**混同しない**。

## 1. コードのレビュー → 「批判的レビューを、重大な指摘が無くなるまで」
- 対象: 計算ロジック・実装・CI・デプロイ等、**正しさが問える**もの。
- 方式: バグ・誤計算・型崩れ・破損 を**批判的に洗い、重大な指摘がゼロになるまで修正**。
- 強制: CI（計算ユニットテスト＋テスト網羅チェック＋型チェック＋ビルド＋デプロイ検証）。緑でなければ公開しない。

## 2. 企画・調査・プランのレビュー → 「批判的レッドチーム → 判定 → 最小検証」
プランは「正しい/間違い」ではなく「**良い賭けか**」。だから *ゼロになるまで磨く* のは不適切
（磨いた駄案は駄案のまま）。代わりに **批判的に潰し、GO/見送り/却下を決め、当たりは最小コストで実地検証**する。
**市場データが最終審判**。

### レッドチームの観点（各提案をこの目で叩く）
1. **需要**は本物で検証可能か（検索数・証拠）。一過性のバズでないか。
2. **勝てるか**：新規・無名ドメインで本当に上位/シェアを取れるか（飽和・大手独占＝money-tools の YMYL 問題を繰り返さない）。
3. **収益モデル**：現実的な流入×転換で成立するか。**EVを数字で**出す。盛らない。
4. **集客**：どう人を連れてくるか（SEOなら期間と難度／予算やフォロワーが要るなら明記）。
5. **コスト**：構築・運用（トークン＋運営者の手間）に見合うか。
6. **リスク**：法令・各規約・YMYL・スパム・無断転載 に触れないか（あれば即 KILL）。
7. **差別化**：AIが量産/自動化で勝てる本当の edge があるか。"me too" でないか。
8. **最小検証**：最も安く速く「需要があるか」を試す方法と、**撤退基準(kill criteria)**。

### 判定（必ずどれか1つ）
- **GO**：最大の前提リスク1つ＋それを試す最小実験＋撤退基準を明記して実行へ。
- **REVISE**：具体的な穴を指摘して差し戻し（次回 戦略家が修正 or 却下）。
- **KILL**：理由を明記して捨てる（`strategy-log.md` に残す）。

### 誰がレビューするか（多層）
- **戦略家の自己レッドチーム**：提案を書く前に自分で叩き、弱いものは捨てる＝一次フィルタ。
- **ビルダーの着手前レビュー**：実装に入る前にこの観点で叩く。致命傷なら `needs-revision` を付けて**作らずコメントを残す**＝独立チェック。
- **運営者(あなた)**：`new-project`/`needs-operator`（口座・決済・新ドメインが要る大型）は**最終GOを人が判断**。
- **データ**：公開後は計測が審判。安く出して、伸びなければ畳む。

### 原則
- **高コストな賭けほど厳しく事前レビュー**。安く試せる実験は "レビューで殺しすぎず" 出して実地で判定する。
- 事前レビューに時間/トークンをかけ過ぎない。完璧な計画より、**安い実験 × 速い学習**。
